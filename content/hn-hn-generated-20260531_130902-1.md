# Show HN: Research Server Patterns – Claude Code persistent memory and email triggers

Claude Code forgets everything between sessions. I was spending 5–10 minutes re-explaining context at the start of every session across multiple projects. Over a few months of daily use, that's a real tax. Research Server is my fix.

The core is an MCP server backed by SQLite FTS5. Sessions call `session_save()` before they end and `session_summary()` when they start. The free recipe is a standalone stdio server — five tools, one SQLite file, five minutes to add via `claude mcp add`. That part is at https://github.com/shukumei7/research-server-patterns.

The more interesting piece is the email trigger system. I forward an invoice to a specific inbox; a Claude Code agent wakes up, processes it using a prompt template, and replies with the result. The mechanism is just IMAP polling every 60 seconds and `claude -p`. No webhooks, no Lambda, no queue. Rough but it works for low-frequency tasks where a 60-second delay is fine.

I also run a nightly cron that drafts SR&ED diary entries from git commits and terminal history. Canadian founders doing SR&ED claims need contemporaneous documentation — this solves the "you reconstructed it post-hoc" risk from CRA without requiring manual discipline.

Limitations worth stating: it requires a persistent process running locally or on a VPS, IMAP polling means ~60s latency on email triggers, and the SQLite backend is intentionally single-machine. This isn't designed to scale; it's designed to work reliably for one founder or a small team.

The free recipe covers memory persistence. The full setup — cron orchestrator, IMAP triggers, terminal, all configured — is $2,500 CAD flat if you'd rather not wire it up yourself.
